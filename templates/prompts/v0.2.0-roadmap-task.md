Copy this prompt into Codex or Claude Code when implementing the v0.2.0 roadmap.

```text
You are a senior CLI/package engineer working on @dowonk-7949/llm-wiki-standard.

Workspace:
C:\Users\samkj\Documents\private-project\llm-wiki-standard

Primary goal:
Implement the v0.2.0 roadmap items that make LLM-WIKI useful after the initial wiki is already created and enriched.

Read first:
- README.md
- README.ko.md
- ROADMAP.md
- RELEASE_CHECKLIST.md
- src/cli.js
- src/commands.js
- src/frontmatter.js
- src/template-renderer.js
- templates/core/wiki-document.md
- tests/frontmatter.test.js
- tests/verification.test.js

Non-negotiable principles:
- Keep existing CLI behavior backward compatible unless tests explicitly change.
- Do not overwrite existing project wiki docs or adapter files without explicit write options.
- CLI-created or agent-edited docs remain needs_review.
- verified status is human-approved only.
- Keep README.md as the default English entrypoint and README.ko.md as the Korean entrypoint.
- Update English and Korean README content together when user-facing behavior changes.
- Use apply_patch for manual edits.
- Run tests after changes.

Implement in this order:

1. Add repeatable task prompt generation.
   - Add public command shape: llm-wiki prompt --task <feature|fix|refactor|docs-sync|okf-extract>.
   - The prompt command should support --cwd, --type, --profile, --agent, --format, and --out when useful.
   - Add command help and parseArgs validation.
   - Add tests for parsing, output text, JSON shape, and --out report writing.

2. Add feature/fix/refactor task prompts for post-wiki development.
   Each generated prompt must instruct the agent to:
   - Read docs/llm-wiki/index.md first.
   - Locate related domain, API, component, architecture, workflow, and decision documents.
   - Inspect actual source files before making claims or edits.
   - Produce a short implementation plan.
   - Make the code change.
   - Update affected LLM-WIKI documents in the same task.
   - Append docs/llm-wiki/log.md in append-only style.
   - Keep AI-edited documents as needs_review.
   - Run relevant tests or explain why they were not run.

3. Add API service inventory expectations to domain documentation.
   Update generated domain-oriented templates and prompts so domain docs include:
   - API service name.
   - Endpoint or client module.
   - HTTP method or call signature.
   - Request params/payload.
   - Response shape.
   - Auth/session/token/cookie dependency.
   - Error handling.
   - Retry/timeout behavior.
   - Cache/state update behavior.
   - Related UI/domain workflow.
   - source_files evidence.

4. Add docs-sync prompt.
   - It should focus on detecting changed code, finding affected wiki docs, updating stale docs, and appending docs/llm-wiki/log.md.
   - It must avoid unrelated code edits.

5. Add OKF extraction prompt.
   - Implement prompt --task okf-extract as a prompt workflow, not automatic extraction.
   - It should follow OKF v0.1: Markdown plus YAML frontmatter, required type, optional aliases/tags, and body wiki links like [[Concept Name]].
   - Extracted documents must remain needs_review when stored in an LLM-WIKI project.

6. Update documentation.
   - README.md and README.ko.md must both document llm-wiki prompt.
   - ROADMAP.md should mark implemented items with Status: implemented.
   - RELEASE_CHECKLIST.md should include prompt command verification.

7. Verify.
   Run:
   - node --test tests/*.test.js
   - node bin/llm-wiki.js validate-frontmatter
   - node bin/llm-wiki.js validate-frontmatter --strict
   - node bin/llm-wiki.js prompt --task feature --agent codex
   - node bin/llm-wiki.js prompt --task docs-sync --agent codex
   - node bin/llm-wiki.js prompt --task okf-extract --agent codex
   - npm pack --dry-run

Expected outcome:
- The package can generate initial setup/handoff prompts and post-wiki feature/fix/refactor/docs-sync prompts.
- Domain documentation explicitly captures API service usage.
- README.md and README.ko.md stay aligned.
- Tests pass.
```
